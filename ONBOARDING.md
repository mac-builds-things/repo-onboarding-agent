# Onboarding Report Format

This document describes the structure and content of every section in an Onboarding Report generated by `repo-onboarding-agent`. Reports are written as Markdown and optionally serialized as JSON for machine consumption.

---

## Report sections

### 1. Project Overview

**What it contains:**
- One-paragraph description of what the project does, inferred from README, docstrings, and package metadata
- Primary language(s) and runtime
- License
- Last active commit date and approximate commit velocity (commits/week over last 90 days)
- Rough size signal: lines of code, file count, directory depth

**How it's generated:**
- README.md is read and summarized (LLM pass if available, else first 400 characters)
- `pyproject.toml`, `package.json`, `Cargo.toml`, etc. supply name, description, license
- `git log` provides recency and velocity signals

---

### 2. Tech Stack

**What it contains:**
- Languages (primary and secondary, with file-count rationale)
- Web framework (if any): FastAPI, Django, Express, Rails, etc.
- Database(s): detected from dependency names and config files
- Background job system (Celery, Sidekiq, BullMQ, etc.)
- Caching layer (Redis, Memcached)
- Testing frameworks
- Build / packaging tools
- Infrastructure hints (Dockerfile, `fly.toml`, `vercel.json`, Helm charts, etc.)

**How it's generated:**
- Dependency files are parsed: `requirements.txt`, `pyproject.toml`, `package.json`, `Gemfile`, `go.mod`, `Cargo.toml`
- Config files are scanned for framework signatures
- File extensions give language breakdown

---

### 3. Directory Map

**What it contains:**
- Annotated tree of top-level and second-level directories (depth 2 by default)
- One-line purpose annotation for each significant directory
- Files that are important enough to call out at the root level (e.g., `Makefile`, `docker-compose.yml`, `schema.sql`)

**How it's generated:**
- Directory tree is walked
- Known patterns are recognized (e.g., `migrations/` → database migrations, `tests/` or `spec/` → test suite, `docs/` → documentation)
- Directories with very high file counts are noted

---

### 4. Key Commands

**What it contains:**
- Every runnable command found in the repo, organized by category:
  - **Install**: dependency installation commands
  - **Dev**: local development server commands
  - **Test**: test runner commands (with any common flags)
  - **Lint / Format**: linting and formatting commands
  - **Build**: compile / bundle commands
  - **Database**: migration, seed, and reset commands
  - **Deploy**: deployment commands

**How it's generated:**
- Parsed from: `Makefile` (targets + comments), `package.json` (`scripts`), `pyproject.toml` (`[tool.taskipy]` or `[tool.poe.tasks]`), `Justfile`, `Taskfile.yml`, `CONTRIBUTING.md` code blocks
- Commands are de-duplicated and classified by name pattern and content

---

### 5. Test Setup

**What it contains:**
- Test framework and version
- How to run the full test suite
- How to run a single test file or test case
- Test database setup requirements (if any)
- Coverage command
- Known slow or flaky tests (if detectable from CI config or test names)
- Current test count (approximate, from file scan)

**How it's generated:**
- `pytest.ini`, `jest.config.*`, `.rspec`, `vitest.config.*` etc. are read
- CI config (`.github/workflows/`, `.circleci/`) is scanned for the canonical test invocation
- Test files are counted and test function/describe blocks are tallied

---

### 6. Risky Areas

**What it contains:**
- A prioritized list of files or modules that warrant extra care, each with:
  - File path
  - Risk category (see below)
  - Risk reason (human-readable explanation)
  - Recommended action (read-before-touch, add tests first, consult owner, etc.)

**Risk categories:**
| Category | Detection signal |
|---|---|
| `auth` | File path or import contains: auth, login, token, jwt, session, oauth, password, credential |
| `billing` | File path or import contains: billing, payment, stripe, subscription, invoice, charge |
| `crypto` | File uses: hashlib, cryptography, bcrypt, hmac, secrets, fernet |
| `no-tests` | Module has no corresponding test file and is not a utility/helper |
| `high-churn` | File changed in >10 commits in the last 90 days (git log heuristic) |
| `large-file` | File is >500 lines (Python/JS) or >1000 lines (generated) |
| `complex-deps` | Module imports >15 other internal modules |
| `incident-touched` | File was modified in a commit whose message contains: hotfix, incident, revert, critical, emergency |
| `god-object` | Single file contains >10 classes or >20 functions |
| `env-access` | File reads from environment variables directly (os.environ, process.env) in many places |

**How it's generated:**
- Static analysis pass over all source files
- Git log inspection for churn and incident signals
- Import graph construction for dependency complexity

---

### 7. Entry Points

**What it contains:**
- The main execution paths into the system:
  - **API routers / route files**: where HTTP endpoints are defined
  - **CLI entry points**: `console_scripts` in `pyproject.toml`, `bin/` directory, Click/Typer app roots
  - **Background workers**: Celery tasks, cron definitions, queue consumers
  - **Main module**: `if __name__ == "__main__"` files, `main.py`, `app.py`, `server.py`
  - **Scheduled jobs**: cron expressions, APScheduler definitions

**How it's generated:**
- `pyproject.toml` `[project.scripts]` and `[project.entry-points]` are read
- Files named `main.py`, `app.py`, `server.py`, `wsgi.py`, `asgi.py` are flagged
- FastAPI/Flask/Django router registration patterns are detected
- Celery `@app.task` and `@shared_task` decorators are found

---

### 8. First Good Tasks

**What it contains:**
- 3–5 suggested tasks a new contributor could do to build familiarity without high risk:
  - Task title
  - Why it's a good first task (isolated, well-scoped, low blast radius)
  - Relevant file(s)
  - Estimated effort (S / M / L)

**Selection criteria:**
- Functions or modules with a `# TODO` comment and no corresponding test
- Small, isolated utility modules with no test file
- Clearly-named issues in `TODO.md` or inline comments
- Linting violations that are mechanical to fix
- Missing docstrings on public API functions (if the project has a docstring convention)
- Type annotations missing on small utility functions

**How it's generated:**
- TODO comment scan across all source files
- Cross-reference with test coverage map to find untested-but-small functions
- LLM pass (optional) to score task quality and write the "why" explanation

---

### 9. Open Questions

**What it contains:**
- Things the agent couldn't determine confidently and that a new contributor should clarify early:
  - Examples: "No `.env.example` found — unclear what environment variables are required", "Two database URLs referenced — unclear which is primary", "No deployment config found — unclear how this reaches production"

**How it's generated:**
- Generated as a byproduct of the other analysis passes: any required-but-missing signal is logged as an open question
- LLM pass (optional) to frame questions helpfully

---

## Report metadata

Every report includes a header block:

```
repo:         <repo name>
path:         <absolute path analyzed>
generated_at: <ISO 8601 timestamp>
agent_version: <version string>
git_sha:      <HEAD commit SHA at time of analysis>
```

This allows reports to be stored, versioned, and compared over time.

---

## Regeneration cadence

Recommended triggers for re-running the agent:
- On merge to the main branch (CI integration)
- Weekly scheduled run
- Before starting a significant new feature
- After a large refactor or dependency upgrade

Sections that change frequently (risky areas, first good tasks) benefit from more frequent regeneration. Sections that change slowly (directory map, tech stack) can be cached aggressively.
